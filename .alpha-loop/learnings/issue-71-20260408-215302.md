## What Worked
- Custom ESLint rule (`no-hardcoded-colors`) correctly targets hex colors in `className` strings while ignoring legitimate uses in metadata objects and JS constants
- Code review caught a critical wiring gap — the shared config was never imported by `apps/web`, meaning the rule couldn't actually prevent regressions. Fixed before completion.

## What Failed
- The ESLint config was not wired into any consuming app initially, which would have silently defeated the purpose of the rule. Caught only by code review, not by tests.

## Patterns
- **Lint rule + integration test together**: Writing a unit test for the rule itself is necessary but insufficient — verifying the rule is actually loaded by consuming apps is equally important
- **Scope className/style only**: Restricting detection to `className` and `style` attributes avoids false positives on metadata (`themeColor`) and JS constants (`CONFETTI_COLORS`), keeping signal-to-noise high

## Anti-Patterns
- **Testing the rule without testing the wiring**: A passing unit test for an ESLint rule means nothing if no app's ESLint config imports it — always verify end-to-end that the rule fires in the actual lint pipeline
- **Assuming shared config is consumed**: Adding rules to `@repo/config` doesn't guarantee any app uses that config; check for an `eslint.config.mjs` (or equivalent) that imports it
