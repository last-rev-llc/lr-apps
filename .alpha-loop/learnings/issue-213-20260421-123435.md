## Patterns
- When seeding test data into tables with FK references to `auth.users`, call `admin.auth.admin.createUser({id, email, email_confirm})` first (idempotent) and throw on upsert errors rather than ignoring them â€” silent seed failures make RLS tests pass vacuously.
- CI checks that depend on a live service (Supabase, DB, etc.) must run in a job that actually starts that service; wiring them into a generic `lint` script that runs in a service-less job creates a guard that always passes.
- When extending a helper like `requireAppLayoutAccess` from `void` to `AccessResult | null`, prefer narrowing at call sites (`if (!access) return ...`) over `!` non-null assertions so future `publicRoutes` adoption doesn't introduce runtime crashes.

- Adding non-null assertions (`access!.user.id`) at call sites of a helper whose return type is intentionally nullable â€” defeats the type system's purpose.
- Documenting a CI guard in CLAUDE.md ("the `lint:registry` CI check fails the build otherwise") before verifying that the job actually exercises the check end-to-end.
