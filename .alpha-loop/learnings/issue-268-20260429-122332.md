## Patterns
- Reusing `public.set_updated_at()` across sibling migrations (clients → client_sites → site_metadata) keeps trigger logic DRY; the first migration in the chain defines it with `create or replace`, later ones just attach triggers.
- Service-role-write tables follow the pattern: RLS enabled + single `select` policy on `auth.role() = 'authenticated'` + no insert/update/delete policies (service role bypasses RLS).

## Anti-Patterns
- Down migrations that drop a shared function (`public.set_updated_at()`) can break sibling tables if rolled back out of order. Reverse-order rollback is safe, but isolated rollback of an early migration would fail. Worth a comment in the down file or scoping the drop to the first-defining migration only.
