## Patterns
- AI server actions: `generateObject` + zod schema → derive composite/derived fields locally → persist atomically scoped by `(id, user_id)` → wrap in `withSpan` → gate via `hasFeatureAccess` → fall back to deterministic stub when API key missing.
- Tag custom server-action errors (`RateLimitedError`, `FeatureAccessError`) with a stable `code` field, not just class identity, so client-side `instanceof` checks survive Next.js's production error sanitisation.

## Anti-Patterns
- Shipping a server action without wiring it to its UI exerciser in the same PR: the `✨ Plan & score with AI` button is still a TODO no-op (`ideas-app.tsx:327`), so the action has no end-to-end smoke test in this batch even though all listed AC pass.
- Relying on `instanceof CustomError` across the server/client boundary in Next.js — the class identity does not survive serialization unless the error carries a discriminator field.
