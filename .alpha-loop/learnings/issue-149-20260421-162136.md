## What Worked
- All 6 acceptance criteria met: `/api/health` returns 200/503 with JSON status, checks Supabase + Stripe, includes version/commit, and has tests for healthy + degraded states.
- Tests passed on first run (0 retries); 838 web tests + 70 auth + 49 billing all green with full turbo cache.

## What Failed
- Review caught logger using `process.stdout.write`/`process.stderr.write`, which are undefined in Next.js Edge Runtime — would have crashed `/api/vitals` and `proxy.ts` auth callback in production.
- WebVitalsReporter read `appSlug` from a `data-app-slug` attribute that nothing in the codebase sets, so per-app vitals tagging was silently broken.
- Scope creep: issue asked for a health endpoint, but the diff also touched checkout logging, vitals, Sentry wiring, and `.gitignore` — bundling unrelated work into one issue.

## Patterns
- Route handler pattern: thin `route.ts` that delegates to a pure `lib/*.ts` module enables easy mocking (`vi.mock("@/lib/health-checks")`) and keeps HTTP concerns separate from check logic.
- Health endpoint response shape: `{ status, version, commit (short), deploymentId, env, timestamp, checks[] }` with `Cache-Control: no-store` and 503 on degraded — standard and Vercel-friendly.
- Mock `@repo/logger` at module level to spy on structured log calls without depending on the logger's transport.

## Anti-Patterns
- Writing a logger that depends on `process.stdout`/`process.stderr` without checking Edge Runtime compatibility — Next.js edge only polyfills `process.env`. Use `console.*` for cross-runtime logging.
- Reading config from DOM dataset attributes (`document.documentElement.dataset.appSlug`) that no code actually sets — derive from a reliable source (e.g., `pathname`) instead.
- Bundling unrelated changes (logging refactor, Sentry init, vitals route, `.gitignore`) into a single "add health check" issue makes review harder and hides regressions.

## Suggested Skill Updates
- Add to `code-review` skill: check any `@repo/logger` or similar logger for Edge Runtime compatibility — flag direct use of `process.stdout`/`process.stderr`, `fs`, or Node-only APIs in code that may run in edge routes (`proxy.ts`, routes with `export const runtime = "edge"`).
- Add to `testing-patterns` skill: when a route depends on runtime env (Edge vs Node), add an explicit test or type check that exercises the edge path, not just the Node path — unit tests running in Vitest/Node won't catch `process.stdout` being undefined in edge.
