## What Failed
- `pnpm lint` still fails with 22 errors across ~20 files — all pre-existing unused vars/imports that the agent didn't fully clean up
- The PR cannot actually pass its own CI pipeline as shipped, contradicting the "All checks pass on current main branch" acceptance criterion

## Patterns
- When adding a CI lint step to a codebase that never enforced linting, run the linter first and fix all errors (or configure exclusions) before declaring success
- Switching from `next lint` to `eslint .` broadens scope significantly — expect new violations from files Next.js wasn't scanning

## Anti-Patterns
- Declaring CI setup "complete" without verifying the pipeline passes end-to-end; the lint step fails, meaning PRs would be blocked immediately
- Fixing some lint errors but not all — partial cleanup leaves the pipeline in a broken state that's worse than not adding the check

## Suggested Skill Updates
- Add a verification gate to implementation planning: when the task involves CI/linting, the agent must run the full pipeline locally and confirm zero failures before marking complete
- Add guidance that switching lint tooling (e.g., `next lint` → `eslint .`) requires a full audit of new violations, not just fixing the ones encountered during development
