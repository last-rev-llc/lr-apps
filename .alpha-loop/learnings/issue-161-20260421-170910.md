## Patterns
- Expose a test-only `_resetRateLimitStore()` from modules holding in-process state so integration tests can isolate runs without time mocking.
- Token-bucket keyed by `x-forwarded-for` IP with per-route config (limit, windowMs) applied at the proxy layer before downstream handlers.
- Double-submit CSRF cookie (readable by JS, SameSite=Lax, Secure in prod) paired with `x-csrf-token` header check; skip for webhook routes that use signature verification instead.

## Anti-Patterns
- In-memory rate-limit store does not survive across serverless instances; effective limit becomes N x configured. Must be documented or migrated to Redis/Upstash before relying on published per-IP figures.
- Accepting but ignoring function parameters (e.g., `BuildCspOptions.reportOnly` with `void opts;`) leaves dead API surface — either wire it up or remove it.
- Bundling unrelated concerns (CSRF, audit logging, validation refactor) into a single rate-limiting issue makes review harder and risks missing spec items from the parent issue.
