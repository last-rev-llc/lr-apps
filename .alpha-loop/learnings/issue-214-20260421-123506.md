## What Worked
- All 762 web tests and package tests passed on first run with no test-fix retries needed.
- Review caught two real CI gaps: missing auth.users seeding in RLS tests (FK violations would mask RLS behavior) and lint:registry silently skipping in CI because Supabase wasn't running in the lint job.
- Fixes were applied in-run rather than deferred: createUser calls made idempotent, and a dedicated `Verify registry → DB consistency` step added to the rls-tests job where Supabase is already running.

## What Failed
- Scope creep: the issue asked for hot-path indexes and an `EXPLAIN ANALYZE`-driven migration, but the diff instead contains a CI workflow for RLS tests, layout refactors to `requireAppLayoutAccess`, and billing-flow test additions — none of the acceptance criteria (idx_app_permissions_app_slug, migration with EXPLAIN ANALYZE comments, paired .down.sql) appear in the changes.
- The "end-to-end DB row write via real upsertSubscription" test mocks the service-role client instead of hitting a real DB, so it does not provide the integration coverage it claims.
- Three layouts now rely on `access!.user.id` non-null assertions that would break if a pathname argument and `publicRoutes` are later added.

## Patterns
- When a CI check depends on external services (Supabase, DB), wire it into the job that already provisions those services rather than the generic lint job — silent-skip checks provide zero protection.
- Seed FK-dependent test users via `admin.auth.admin.createUser({ id, email, email_confirm })` with thrown errors on failure, so RLS tests fail loudly instead of passing because seed data never landed.

## Anti-Patterns
- Silent-skip guards in CI scripts (exit 0 when env vars missing) create false confidence — prefer failing loudly or moving the check into a job with the required env.
- Not checking upsert/insert errors in RLS test setup lets SELECT-isolation assertions pass for the wrong reason (0 rows from failed seed vs. 0 rows from RLS denial).
- Non-null assertions (`!`) on helper return values whose null contract depends on caller-supplied arguments — brittle when call sites evolve.
- Acceptance criteria drift: the run shipped unrelated CI/refactor work while none of the stated DB index deliverables landed.
