## Patterns
- **Review-as-verification for CSS migrations**: Unit tests cannot catch color semantic mismatches (pink vs red for "bad"), gradient collapse, or missing theme variant overrides — structured code review is the essential verification step for token work
- **Consistency audit within a PR's own conventions**: The glass-input rgba vs oklch inconsistency was caught by checking the PR against itself, not just against the codebase — PRs that introduce a convention should follow it uniformly

## Anti-Patterns
- **Skipping unmet acceptance criteria without follow-up**: Visual regression baselines were an explicit AC but silently dropped — unmet criteria should block success status or generate a tracked follow-up issue
- **Over-trusting "mechanical" changes**: Labeling token migration as "token-for-token with no behavioral changes" led to skipping verification, yet review found gradient collapse and semantic color mismatches — never assume CSS changes are safe without visual confirmation
