## Patterns
- Icon-only buttons need `aria-label` for both a11y and Playwright `getByRole({ name })` locators — the W3C accessible-name calculation prefers subtree text (the emoji) over the `title` attribute.
- When multiple components on a page render the same pill/label text (e.g. "All"), scope test locators with `.first()`/`.last()` or a parent container, or distinguish via `aria-label`, to satisfy Playwright strict mode.
- `scripts/audit-tokens.ts` only flags Tailwind palette classes and hex/rgb/oklch literals — it does NOT flag `text-white/<opacity>` utilities or `var(--color-X)` inline styles. AC saying "audit passes" is weaker than AC saying "no inline `var(--color-...)`" — both are needed to fully enforce the no-hardcoded-colors rule.

## Anti-Patterns
- Scoping a "token cleanup" issue to a single file (`ideas-app.tsx`) when the same patterns exist in 4+ sibling components in the same directory — leaves a known-incomplete state and forces a follow-up sweep.
- Writing locators against icon-only buttons by accessible-name regex without first verifying the computed name (the `title` attribute does NOT contribute when subtree content exists).

## Suggested Skill Updates
- `test-robustness` / `playwright-cli`: add a rule — "When asserting on icon-only buttons via `getByRole('button', { name: ... })`, the button must have an `aria-label`; `title` is not used for accessible-name when subtree text (including emoji) is present." Reference the Snooze button case as the example.
- `code-review` skill: add a check item — "When a token-cleanup / lint-rule issue names a single file but the pattern recurs in sibling files in the same directory, flag the scope mismatch and recommend either expanding the AC or filing a follow-up issue before merge."
