## Patterns
- When a review issue bundles findings across multiple severity levels, explicitly document which info-severity items are out-of-scope for the AC (hover/token only) so reviewers can see the deliberate deferral rather than assuming oversight.
- Pair a UI-primitive migration with component tests in the same PR — the tests lock in the behavior that `<UIInput>`/`<Badge>` replacements are functionally equivalent to the hand-rolled markup.
- Use markered listings in CLAUDE.md (`<!-- lib-listing:start -->`/`end`) backed by a CI sync script so directory-structure docs can't silently drift.

## Anti-Patterns
- Don't migrate to `@repo/ui` primitives that don't exist yet (e.g., Checkbox/Select) — retain native elements and file a follow-up rather than inventing a partial primitive just to satisfy a "use the design system" instinct.
- Don't delete review-issue JSON files before confirming that every info-severity finding has been either addressed or explicitly deferred in the PR description, otherwise the deferrals become invisible.
