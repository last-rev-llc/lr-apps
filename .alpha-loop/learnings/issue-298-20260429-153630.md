## Patterns
- Reuse the established `lib/rate-limit.ts` helper rather than introducing a new limiter — keeps key conventions consistent across `stripe-webhook`, `ideas.planAndScoreIdea`, etc.
- Compose the rate-limit key as `${userId}:${actionName}` so multiple AI actions per user don't collide on the same bucket
- Throw a named error class (not a string) before the expensive call, then log with `resetAt` — gives the UI a stable handle and ops a queryable signal

## Anti-Patterns
- Relying on `instanceof RateLimitedError` on the client: Next.js sanitises uncaught server-action errors in production, so class identity is lost across the boundary. Tag errors with a stable `code` or message string when client-side branching is required
- Shipping a server action without wiring its UI invoker in the same batch: the action exists but `ideas-app.tsx:327` still has a `// TODO: invoke AI plan action` no-op button. Acceptance criteria were met, but no end-to-end exerciser landed
