## What Failed
- Initial implementation only wired audit logging into billing handlers and skipped the auth0-factory `onCallback` path required by the spec — caught by review, not by tests, because no test asserted `auth.login.succeeded`/`auth.login.failed` events.
- `@repo/billing` tests show repeated `logAuditEvent threw … client.from(...).insert is not a function` warnings: the billing test mocks for the supabase client don't implement `.from().insert()`, so audit calls silently swallow an error in tests. Tests pass because `logAuditEvent` is fail-open, but the warnings indicate the mocks weren't updated alongside the new dependency.

## Patterns
- Fail-open audit logging (catch + log error, never throw) is the right default for cross-cutting observability so a logging bug can't take down billing/auth flows.
- Double-submit CSRF (cookie + `x-csrf-token` header, SameSite=Lax, no HttpOnly) with a webhook-path skip list is a clean pattern for Next.js route handlers and is now exercised in `proxy.integration.test.ts`.
- When adding a helper that touches the DB, update shared test mocks (`@repo/test-utils/mock-supabase`) in the same change so downstream packages don't emit silent fail-open warnings.

## Anti-Patterns
- Implementing a spec bullet ("called from auth and billing handlers") in only one of the two locations and relying on review to catch the gap — every spec-required call site should have an assertion in the corresponding test file.
- Accepting a parameter (`BuildCspOptions.reportOnly`) and immediately discarding it with `void opts;` — either implement it or drop it from the signature; dead parameters mislead callers.
- Letting a new `logAuditEvent` dependency log `client.from(...).insert is not a function` from billing tests instead of extending the supabase mock — silent fail-open in production code shouldn't mean silent broken mocks in tests.

## Suggested Skill Updates
- `testing-patterns`: add a rule that when a spec says "called from X and Y", the test suite must include an assertion (mock spy) for *each* call site, not just one — this would have caught the missing auth-handler wiring before review.
- `code-review`: add a check for "accepted-but-ignored parameters" (e.g. `void opts;`, unused destructured fields) as a low-priority finding worth flagging.
- `testing-patterns`: when a new helper in `@repo/db` is added, update `@repo/test-utils/mock-supabase` in the same PR so dependent packages don't emit fail-open error logs in their test output.
