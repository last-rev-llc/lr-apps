## What Worked
- All 9 packages tested green (838 web tests, 221 ui tests, 70 auth tests) on first try with zero test-fix retries
- Sentry wiring landed cohesively across three runtimes (client/server/edge) with error boundary + global-error.tsx + logger.error forwarding
- Structured logging with `withRequestContext` + requestId added to checkout/webhook routes alongside the Sentry integration

## What Failed
- Logger used `process.stdout.write`/`process.stderr.write`, which are undefined in Next.js Edge Runtime — would have 500'd `/api/vitals` and crashed the auth callback in `proxy.ts`
- `WebVitalsReporter` read `appSlug` from `document.documentElement.dataset.appSlug`, but nothing sets that attribute — per-app vitals tagging was silently broken
- Pre-existing typecheck error in `app/apps/ai-calculator/(protected)/calculator/page.tsx` blocks clean webpack build, leaving bundle baseline numbers TBD (not introduced here)

## Patterns
- When adding a logger used across runtimes, prefer `console.*` over `process.stdout.write` — works in Node, Edge, and browser without polyfills
- Derive contextual metadata (like `appSlug`) from stable sources (URL pathname) rather than DOM data attributes that require separate wiring
- Acceptance criteria requiring external verification (e.g., "errors appear in Sentry dashboard") should be flagged as unverifiable from code review alone

## Anti-Patterns
- Using Node-only APIs (`process.stdout`, `fs`, etc.) in shared library code that runs in Edge Runtime — silently breaks at runtime, not at build
- Reading from DOM data attributes without verifying any code actually populates them — produces undefined values that defeat the feature
- Claiming acceptance criteria met when verification requires runtime artifacts (live DSN, dashboard check) outside the code

## Suggested Skill Updates
- Add to `code-review` skill: check for Node-only APIs (`process.stdout/stderr.write`, `fs`, `Buffer`) in code paths that run in Next.js Edge Runtime (middleware, `runtime = "edge"` routes, `proxy.ts`)
- Add to `code-review` skill: when code reads from `dataset.*` or other DOM attributes, grep the codebase to verify something actually sets the attribute
