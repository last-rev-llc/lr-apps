## Patterns
- When migrating inline UI to shared components, grep the entire monorepo for tests asserting on old CSS classes — upstream component changes (like MoodBadge → StatusBadge) ripple into unrelated test files
- Using `@repo/ui` EmptyState for zero-data states provides consistent UX across apps with minimal code
- Card + CardHeader + CardTitle composition pattern works well for grouped list sections (client grouping headers)

## Anti-Patterns
- Updating a shared sub-component (MoodBadge) without scanning for downstream test assertions caused the first test failure — always search for class-name assertions across the test suite before committing
- Hardcoding color classes in tests couples tests to implementation details; asserting on semantic variant names or data attributes would be more resilient
