## What Worked
- Clean implementation of subscription management UI with server action for Stripe Customer Portal session creation
- Comprehensive test coverage (8 UI state tests, 5 server action tests, 3 portal tests) — all passed first try with zero retries
- Graceful free-tier handling with null/missing stripe_customer_id branching to "Free Plan" + pricing link

## What Failed
- AC partially unmet: "payment method" display was not implemented inline — deferred to Stripe Portal instead. The `Subscription` DB type lacks payment method fields, requiring an additional `stripe.paymentMethods.retrieve` API call that wasn't scoped into the work

## Patterns
- When an AC requires data not in the local DB schema (e.g., payment method from Stripe), flag it early and either expand the data model or document the gap before implementation begins
- Delegating payment method management entirely to Stripe Customer Portal is a valid architectural choice but should be called out as an AC deviation upfront

## Anti-Patterns
- Accepting ACs at face value without verifying data availability in the existing schema — the missing payment method field should have been caught during planning
- Leaving debug artifacts (pricing-error.png) in the repo root untracked
