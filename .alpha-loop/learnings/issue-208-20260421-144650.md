## What Failed
- env.ts `env()`/`parseEnv()` exports are never invoked at runtime — docs claim startup validation but wiring would break existing test suites that don't set all required vars.
- scripts/__tests__/check-migration-pairs.test.ts not picked up by `pnpm test` (scripts/ lacks package.json and isn't in pnpm-workspace.yaml).
- docs/ops/preview-deployments.md incorrectly states the PR comment workflow triggers on `pull_request` (actually `deployment_status`).

## Patterns
- Pair CI lint scripts with both a dedicated pnpm script (`db:check-migration-pairs`) AND inclusion in `pnpm lint` so failures surface in two places.
- When adding Zod env schemas, widen source parameter to `Record<string, string | undefined>` instead of casting to `NodeJS.ProcessEnv` — avoids TS 5 strictness on required fields like NODE_ENV.
- Retroactive companion files (`.down.sql` for every existing migration) should land in the same PR as the lint check that enforces them, so CI passes from day one.

## Anti-Patterns
- Documenting behavior ("validates at startup") without wiring the code path — creates drift between docs and reality.
- Placing test files under `scripts/__tests__/` without a package.json — they silently get excluded from `turbo run test`.
- Using `as NodeJS.ProcessEnv` casts in test fixtures — breaks under TS 5's stricter ProcessEnv typing.

## Suggested Skill Updates
- docs-sync: Add check that documented runtime behavior (e.g., "validates at startup") has a corresponding code import/invocation, not just an exported function.
- testing-patterns: Note that test files outside workspace packages (e.g., `scripts/__tests__/`) won't run under `turbo run test` unless the directory has a package.json or is added to pnpm-workspace.yaml.
