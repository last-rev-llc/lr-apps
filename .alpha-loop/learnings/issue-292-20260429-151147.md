## What Worked
- All three required server actions (`rateIdea`, `toggleHideIdea`, `snoozeIdea`) shipped with zod validation, session-derived `user_id`, and 37 passing test cases covering happy paths and edge cases.
- Cross-issue review caught a real inconsistency: `setIdeaStatus('archived')` was clearing `completedAt` while `archiveIdea` preserved it — the two archive paths were fixed to agree.
- Tests passed on first run with no retry cycle; the in-memory Supabase mock pattern (filter-based builder with `eq`/`single`/`maybeSingle`) handled all action shapes cleanly.

## What Failed
- `toggleHideIdea` ships as SELECT-then-UPDATE (two round-trips), not the single round-trip the AC specified. Reviewer correctly noted supabase-js has no server-side toggle expression — true single round-trip would need an RPC. Acceptance criterion was technically not met but flagged INFO.
- `FEATURE_LABELS['ideas:ai-plan']` was added in `tier-config.ts` but duplicates the existing `FEATURE_TIER` entry in `@repo/billing` — two sources of truth for the same gate, with no current consumer of the new map.

## Patterns
- **Cross-issue AC reconciliation**: when batch-implementing related issues (#292 toggle, #293 archive), check that overlapping behaviors (e.g. `setIdeaStatus('archived')` vs `archiveIdea`) agree — divergent paths to the same end-state silently destroy data.
- **In-memory Supabase mock**: `vi.hoisted` for shared user IDs + filter-array builder with chainable `.eq()/.order()/.single()/.maybeSingle()` is reusable across action test suites and avoids brittle per-call mocks.

## Anti-Patterns
- Claiming "single round-trip" in an AC for an SDK that doesn't support it — either spec the RPC explicitly or relax the AC. The implementation silently fell back to two round-trips with no acknowledgment.
- Adding a new label/config map (`FEATURE_LABELS`) parallel to an existing one (`FEATURE_TIER`) without wiring a consumer or deleting the old one — creates drift risk for zero current benefit.
