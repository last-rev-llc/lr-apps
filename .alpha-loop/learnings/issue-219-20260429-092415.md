## What Worked
- All 5 acceptance criteria implemented in one pass: migration with paired down.sql, `getFeatureFlagValue()` with full resolution chain, exports, unit tests, layout gating, and seed data.
- Test suite passed cleanly across 12 packages (906 web tests, 60 billing tests including new 7-test feature-flags suite) with zero retries.

## What Failed
- Initial implementation of `addUserOverride` in `flags-app.tsx` omitted the optimistic UI update that the sibling handlers (`handleGlobal`/`handleTier`/`handleRemove`) all had — an inconsistency the author missed.
- New test mock builder used untyped objects that triggered `TS2571` in `pnpm typecheck`; type narrowing was needed at the call site.
- Issue #218 AC specified client-side `track('login', ...)` but implementation used server-side `capture()` from the Auth0 callback — behaviorally equivalent but a literal AC deviation that wasn't flagged at planning time.

## Patterns
- Resolution-order primitives (user → tier → global → default) map cleanly to a single SQL query with `ORDER BY` precedence + `LIMIT 1`, avoiding multiple round-trips.
- Pairing each flag mutation handler with a matching optimistic-update + rollback block keeps UI handlers symmetrical and easier to review for omissions.
- Seeding feature flags with `enabled=false` in `seed.sql` lets the gate ship dark and be flipped per-environment without redeploy.

## Anti-Patterns
- Copy-pasting sibling handlers without diffing the optimistic-update logic — easy to drop a step (as happened with `addUserOverride`).
- Untyped mock builders in new tests: write `as` narrowing or proper types up front; `unknown` propagation surfaces only at typecheck, not test run.
- Treating server-side equivalents of client-side ACs as "good enough" without flagging the deviation in the PR — even when the behavior is arguably better, the AC literal-match should be called out.
