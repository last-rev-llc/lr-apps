## What Failed
- Logger initially used `process.stdout.write` / `process.stderr.write`, which are undefined in Next.js Edge Runtime — would have crashed `/api/vitals` and `proxy.ts` auth callback in production.
- `WebVitalsReporter` read `appSlug` from a `data-app-slug` attribute that nothing in the codebase ever sets, silently defeating per-app vitals dimensioning.
- Per-route First Load JS numbers in the baseline doc are TBD because a pre-existing typecheck error in `app/apps/ai-calculator/(protected)/calculator/page.tsx` blocks a clean webpack build.

## Patterns
- When adding logging to code that may run in Edge Runtime (middleware, `proxy.ts`, routes with `export const runtime = "edge"`), use `console.*` rather than `process.stdout/stderr.write` — `console` works in both Node and Edge; `process` is polyfilled to only expose `process.env`.
- Derive `appSlug` from `pathname` (apps live under `/apps/<slug>/...`) rather than relying on a DOM data attribute that would need to be set from a server layout.
- Acceptance criteria that require external system verification (e.g., "errors appear in Sentry dashboard") cannot be confirmed from code review alone — flag them as manual smoke tests rather than marking complete.

## Anti-Patterns
- Writing logger transports against `process.stdout`/`process.stderr` without checking which runtimes the logger is imported into — Edge Runtime silently lacks these and will throw at first call.
- Reading configuration from DOM data attributes without verifying something actually sets them — the failure mode is silent `undefined`, not a visible error.
- Documenting "baseline" numbers as TBD without linking the blocker (the typecheck error) to a tracked follow-up issue.

## Suggested Skill Updates
- Add to `code-review` skill: checklist item for logger/console usage in Edge Runtime contexts — grep for `process.stdout` / `process.stderr` in any package that could be imported from middleware, `proxy.ts`, or a route with `runtime = "edge"`.
- Add to `implementation-planning` skill: when a task depends on external system verification (Sentry, Stripe dashboards, analytics), explicitly split acceptance criteria into "code-verifiable" vs "requires manual smoke test" so agents don't claim success on the latter.
