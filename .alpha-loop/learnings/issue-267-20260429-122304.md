## Patterns
- When introducing a new table that depends on a shared trigger function, define the function in the *first* migration that needs it and have later migrations reuse it via `execute function public.set_updated_at()` — avoids duplicate `create or replace function` calls.
- Quoted camelCase columns (`"clientId"`, `"openTicketCount"`, `"isPrimary"`, `"ticketsLastUpdated"`, `"createdAt"`, `"updatedAt"`) consistently used to match TypeScript/Supabase client expectations without snake_case mapping.
- Composite unique constraint `(user_id, "clientId", url)` plus separate index on `(user_id, "clientId")` — the unique constraint covers exact-match lookups, the index handles per-client list queries.

## Anti-Patterns
- Down migration drops `public.set_updated_at()` even though sibling tables (`client_sites`, future `site_metadata`, etc.) reuse it — a partial rollback of just the `clients` migration would break those tables' triggers. Safe under reverse-order rollback, but the function drop should live in whichever migration is *last* to reference it, not the first.
- The plan referenced "same pattern as ideas" but no `ideas` migration exists in the repo — copying-pattern instructions should cite a verified existing file, not a remembered one.
