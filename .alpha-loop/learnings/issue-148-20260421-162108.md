## What Worked
- All acceptance criteria met on first pass: web-vitals package integration, /api/vitals endpoint with validation, WebVitalsReporter client component, and baseline documentation.
- Full test suite passed (9/9 packages, 838 web tests) with no test-fix retries needed; new route tests cover valid metric, invalid metric name, and malformed body cases.

## What Failed
- Logger used `process.stdout.write`/`process.stderr.write`, which are undefined in Next.js Edge Runtime — would have crashed `/api/vitals` (edge) and `proxy.ts` auth0 callbacks at runtime despite tests passing in Node.
- `WebVitalsReporter` read `appSlug` from `document.documentElement.dataset.appSlug`, an attribute never set anywhere in the codebase — per-app vitals tagging was silently broken.
- Pre-existing typecheck error in `app/apps/ai-calculator/(protected)/calculator/page.tsx` blocked a clean webpack build, so per-route First Load JS baseline numbers are TBD.

## Patterns
- When adding logging to code that may run in Edge Runtime (middleware, proxy, routes without `export const runtime = "nodejs"`), use `console.*` rather than `process.stdout/stderr.write` — console works in both runtimes.
- Derive app context from URL structure (pathname under `/apps/<slug>/`) rather than DOM data attributes that require separate wiring; single source of truth avoids silent undefined fallbacks.
- Node-environment unit tests don't catch Edge Runtime API gaps — need runtime-matched test env or explicit runtime polyfill checks for code marked `export const runtime = "edge"`.

## Anti-Patterns
- Reading config from DOM attributes that no code sets; the fallback path (`undefined`) compiles and runs but breaks the feature silently.
- Using Node-only APIs (`process.stdout.write`) in a `@repo/logger` package intended for use across all runtimes without declaring or enforcing the runtime contract.
- Marking acceptance criteria complete based on unit tests alone when the criteria include "metrics sent to endpoint" — needs at minimum a runtime smoke test or integration test with the actual target runtime.

## Suggested Skill Updates
- `code-review`: Add Edge Runtime check — flag use of `process.stdout`/`process.stderr`/Node-only APIs in files without `export const runtime = "nodejs"`, and in shared packages imported by edge code (middleware, proxy, route handlers).
- `testing-patterns`: Add guidance that route tests for handlers with `export const runtime = "edge"` should set `// @vitest-environment edge-runtime` (or equivalent) or explicitly stub the Edge process polyfill to catch Node-API leaks.
- `code-review`: Add check for DOM-attribute config reads (`document.*.dataset.*`, `getAttribute`) — verify the attribute is actually set somewhere in the rendered tree, not just read.
