## What Worked
- New `@repo/logger` package with structured JSON output integrated cleanly into checkout route, stripe webhook, health, and vitals endpoints; all 11 logger tests and downstream consumers passed first run.
- `withRequestContext` AsyncLocalStorage pattern correctly propagated `requestId`/`route` through nested calls — visible in test output (e.g. `requestId":"cff202aa..."` consistently attached to checkout logs).
- Test mocks for `@repo/logger` (`vi.mock("@repo/logger", () => ({ log: { info: ... } }))`) kept route tests isolated from logger internals.

## What Failed
- Initial implementation used `process.stdout.write`/`process.stderr.write`, which are undefined in Next.js Edge Runtime — would have crashed `/api/vitals` and the proxy auth0 `onCallback` hook in production. Caught only in review, not by tests.
- `WebVitalsReporter` read `appSlug` from `document.documentElement.dataset.appSlug`, an attribute never set anywhere in the codebase — silently broke per-app vitals attribution.

## Patterns
- AsyncLocalStorage-based request context (`withRequestContext({ requestId, route }, async () => ...)`) is the right shape for propagating log fields across async boundaries in Next.js route handlers.
- For cross-runtime (Node + Edge) logging in Next.js, use `console.{log,warn,error}` — they're polyfilled in Edge and route to stdout/stderr correctly in Node. Avoid `process.stdout.write` / `process.stderr.write`.

## Anti-Patterns
- Assuming Node `process` APIs are available in Edge Runtime. Next.js Edge only polyfills `process.env`; `process.stdout`/`process.stderr` are undefined. Verify against `next/dist/esm/server/web/sandbox/context.js` before using `process.*` in shared utilities.
- Reading data from DOM attributes (`document.documentElement.dataset.*`) without verifying a writer exists — if no code sets the attribute, the read silently returns undefined and tests pass.
- Tests that mock the entire logger don't catch runtime-environment bugs in the logger itself. Logger packages need at least one test that exercises the real implementation in an Edge-like environment.

## Suggested Skill Updates
- Add an Edge Runtime compatibility note to `testing-patterns` or a new section in `simplify`: when shared utilities are used from both Node and Edge code paths (route handlers, middleware, proxy), reject `process.stdout`/`process.stderr`/`fs`/`path` and prefer `console.*` or Web APIs.
- Add to `code-review`: when a client component reads `document.*.dataset.X`, grep for a writer of that attribute; flag as a bug if none exists.
