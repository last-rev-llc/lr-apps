## Patterns
- `validateJson(req, schema)` returning a discriminated union (`{ ok: true, data } | { ok: false, response }`) lets routes early-return the 400 without try/catch around parsing.
- Returning `{ error: "invalid_input", issues: ZodError.issues }` gives clients machine-readable field paths while keeping the top-level error code stable.
- In-memory rate limiter exposes `_resetRateLimitStore()` for tests — required to keep `proxy.integration.test.ts` deterministic across describe blocks.

## Anti-Patterns
- Inline Zod parsing in `app/api/vitals/route.ts` instead of the shared `validateJson` helper — creates two validation styles in the same codebase. Prefer the helper unless the response contract genuinely differs (sendBeacon needs body-less 204).
- Accepting parameters that are immediately discarded (`buildCspOptions.reportOnly` → `void opts;`). Either implement or remove; dead parameters mislead callers.
- Per-instance in-memory rate limiting silently multiplies limits by replica count on serverless. Acceptable for v1 but the limit value in the response header becomes a lie under horizontal scaling.

## Suggested Skill Updates
- `security-analysis`: add a check for "validation helper consistency" — flag routes that re-implement Zod parsing inline when a shared `validate*` helper exists in `lib/`.
- `code-review`: add reminder to verify cross-cutting requirements (audit logging, rate limit, CSRF) are applied to *all* handlers in the spec, not just the ones with the most obvious surface area — the auth callback miss would have been caught by a "spec coverage matrix" pass.
