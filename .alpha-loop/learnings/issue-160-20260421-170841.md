## Patterns
- Double-submit CSRF: cookie without HttpOnly (client reads it) + `x-csrf-token` header compare, `SameSite=Lax`, `Secure` in prod, skip list for webhook routes.
- Test isolation for in-memory rate limiter: export a `_resetRateLimitStore()` helper and call it in `beforeEach` to prevent cross-test bucket leakage.
- `.csp-defaults.json` as a single source of truth for CSP directives, consumed by `lib/csp.ts` and wired into `proxy.ts` response headers.

## Anti-Patterns
- Accepting a parameter (`reportOnly`) then discarding it with `void opts` — either wire it through to change the header name (`Content-Security-Policy-Report-Only`) or drop the param. Dead params lie about the API surface.
- Relying on an in-process `Map` for rate-limiting in a serverless/multi-instance deployment — effective limit becomes `N × configured` where N is instance count. Document this or migrate to a shared store before publishing the limit figure.
- Inline Zod validation in one route while a shared `validateJson` helper is documented for the rest — drift between docs and code confuses future readers.

## Suggested Skill Updates
- `security-analysis`: add a checklist item for double-submit CSRF implementations — verify cookie is NOT HttpOnly (by design), `SameSite` is set, webhook/callback routes are in the skip list, and the header name matches what the client sends.
- `testing-patterns`: note the in-memory-store reset pattern (`_resetFooStore()` exported from the module, called in `beforeEach`) for any module with module-level mutable state (rate-limiters, caches, counters).
- `code-review`: flag parameters that are accepted but immediately discarded (`void opts`, `void param`) as a review smell — either wire them through or remove them.
