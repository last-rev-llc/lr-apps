## Patterns
- Root-level pnpm script → workspace-internal seed module is a clean way to expose dev tooling without polluting any single package's public API.
- Pairing service-role writes with idempotent upserts gives a re-runnable seed without DELETE-then-INSERT churn or FK ordering complexity.
- Reusing `E2E_TEST_USER_ID` env var ties dev seed data to the same identity used by E2E tests, avoiding divergent fixtures.

## Anti-Patterns
- Avoid hardcoding test user UUIDs/emails in the seed script — env-var-driven keeps the script reusable across environments and avoids leaking PII into git.
- Don't use the anon client for seeding — RLS policies will silently drop writes; service role is required and the script must enforce/document this.
