## Patterns
- Gate enforcement at the layout level (not individual pages) is the correct pattern — prevents bypass and keeps page components focused on their domain
- Mocking `hasFeatureAccess` in layout tests to cover both gated/ungated paths provides good coverage without needing real subscription state
- Server actions (`createPortalSessionAction`) with proper auth checks before calling billing APIs is a solid pattern for Stripe portal integration

## Anti-Patterns
- Hardcoding tier requirements in layouts when the app registry already has this data — use a single source of truth to avoid drift
- Committing debug artifacts (screenshots) — add common debug file patterns to `.gitignore`
- Removing existing tests during a refactor without explicit justification — flag removals in PR description
