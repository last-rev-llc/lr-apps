## What Failed

## Patterns
- Pair every new Zod env schema addition with a test file that round-trips through `parseEnv(Record<string, string | undefined>)` rather than casting to `NodeJS.ProcessEnv` — survives stricter TS lib updates
- Staging Stripe = test-mode keys (no separate account); document this explicitly in `.env.staging.example` to prevent future "do we need staging Stripe keys?" confusion
- Add script-level checks (like `check-migration-pairs.ts`) as both a standalone `pnpm` target AND a CI step, since `scripts/` lacks a package.json and is skipped by `turbo run test`

## Anti-Patterns
- Claiming runtime behavior in ops docs ("validated at process startup") without actually wiring the validator into a module that runs at startup
- Using `as NodeJS.ProcessEnv` casts in tests — breaks silently when TS lib tightens the `ProcessEnv` contract (e.g. required `NODE_ENV`)
- Adding test files under `scripts/__tests__/` without workspace membership — they never run in CI and create false coverage confidence
