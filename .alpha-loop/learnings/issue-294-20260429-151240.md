## Patterns
- When implementing a status enum that includes both a "completed" terminal state and an "archived" terminal state, decide up front whether `archived` preserves or clears `completedAt` — and apply the rule to **every** action that writes that status, not just the dedicated archive action.
- Co-locating the server-action mock builder (chainable `.eq().order().single()` shape) inside the test file using `vi.hoisted` for IDs gives a fast in-memory store that mirrors Supabase's PostgREST builder semantics.

## Anti-Patterns
- Two server actions that both transition a row into the same terminal state writing different field sets is a latent inconsistency bug — even if both pass their own tests in isolation, the user observes divergent behavior depending on which UI surface they used.
- Duplicating a feature-gating constant in two packages (`FEATURE_LABELS` in tier-config + `FEATURE_TIER` in billing) sets up silent drift; either remove the unused export or have one re-export the other.

## Suggested Skill Updates
- `code-review/SKILL.md`: add a "Cross-action terminal-state consistency" check — for any enum field where multiple server actions can write the same value (e.g., `status='archived'`), grep for every writer and verify they touch the same companion fields (`completedAt`, `archivedAt`, etc.).
