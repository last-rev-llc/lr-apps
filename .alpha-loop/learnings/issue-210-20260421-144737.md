## What Worked
- All 5 acceptance criteria addressed on first pass with no test-fix retries; full test suite (8 packages, 1200+ tests) passed with turbo cache hits.
- Review caught a TS 5 typecheck regression in `apps/web/lib/__tests__/env.test.ts` (`as NodeJS.ProcessEnv` casts) and fixed it by widening `parseEnv`'s parameter to `Record<string, string | undefined>` — behavior-preserving fix.
- Scope was held: reviewer explicitly chose NOT to wire `env()` into startup because it would break existing test suites, and documented that as a separate follow-up rather than expanding the PR.

## What Failed
- `docs/ops/environments.md` claims env validation runs "at process startup," but `env()`/`parseEnv()` from `apps/web/lib/env.ts` are never imported or invoked — the documented guarantee does not hold at runtime.
- `docs/ops/preview-deployments.md` says the preview-comment workflow triggers on `pull_request`, but the actual workflow uses `deployment_status` — doc/code drift shipped.
- Scope creep: issue #210 is disaster-recovery docs, but the diff also added `.env.example`, `.env.staging.example`, a new CI `preview-comment.yml` workflow, migration-pair lint wiring, and `.alpha-loop.yaml` ops pointers — these belong to adjacent issues, not #210.

## Patterns
- When adding Zod-validated env parsing, type the source parameter as `Record<string, string | undefined>` rather than `NodeJS.ProcessEnv` — avoids TS 5's stricter `NODE_ENV` requirement at test sites and `process.env` still assigns cleanly.
- Acceptance criteria can be satisfied by "schema exists + is tested" without wiring it into the runtime hot path; call out the runtime-integration gap explicitly as a follow-up rather than smuggling it into the same PR.

## Anti-Patterns
- Writing docs that describe aspirational behavior ("validates at process startup", "triggers on pull_request") before the code actually implements it — the two drifted in the same PR.
- Bundling unrelated operational scaffolding (env examples, CI workflows, migration lint) into a disaster-recovery-docs issue. If the review hadn't caught the env typecheck break, this would have been a CI failure on a PR whose title doesn't mention env changes at all.

## Suggested Skill Updates
- `code-review`: add a check that user-facing documentation claims about runtime behavior (e.g., "validates at startup", "runs on X event") are grounded in an actual import/call site or workflow trigger in the same diff — flag as drift when the doc describes behavior the code doesn't enforce.
- `implementation-planning`: when an issue is scoped to docs-only, planning output should explicitly call out that code/config changes belong in separate issues; current run shipped 5+ unrelated files (`.env.example`, `.env.staging.example`, `preview-comment.yml`, migration-pair CI step) inside a docs issue.
