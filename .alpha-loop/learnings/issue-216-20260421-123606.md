## What Worked
- All 8 packages' tests passed (cached) on first run with no retries needed.
- Review caught two genuine CI gaps (missing auth.users seed, lint:registry silently skipping) and both were fixed in-run.
- Wiring `lint:registry` into the `rls-tests` job (which already runs `supabase db reset`) gave the consistency check real teeth.

## What Failed
- Initial RLS test setup omitted creating `auth.users` rows, so FK-dependent seeds would have silently no-op'd in CI and the audit_log INSERT test would have failed with a FK violation rather than testing RLS.
- `lint:registry` was added to `pnpm lint` but the `ci` job has no Supabase running, so the check exited 0 unconditionally until moved to the `rls-tests` job.
- New "end-to-end" billing-flows test still mocks the service-role client instead of hitting a real DB, so it doesn't truly satisfy the integration intent.

## Patterns
- When a CI script depends on a live external service (Supabase), wire it into the job that already provisions that service rather than the generic lint job.
- Seed `auth.users` via `admin.auth.admin.createUser({ id, email, email_confirm: true })` (idempotent) before any test that touches FK-referencing tables.
- Surface seed/upsert errors by throwing — silent `error: null` checks let RLS tests pass for the wrong reason (0 rows because nothing seeded).

## Anti-Patterns
- Adding a "consistency check" to `pnpm lint` without verifying its dependencies (DB connection, env vars) are present in the CI job that runs lint — produces a check that always passes.
- Non-null assertions (`access!.user.id`) on a helper whose return type includes `null` for a documented branch — brittle if the call site later adds the parameter that triggers the null branch.
- Labeling a test "end-to-end DB row write" while still mocking the DB client — misleading naming hides actual coverage gaps.
