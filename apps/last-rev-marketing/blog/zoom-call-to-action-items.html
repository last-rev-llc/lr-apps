The Pipeline: Five Stages in 60 Seconds

Here's the full architecture. Each stage is automated and event-driven — no human intervention required after the Zoom call ends.

Zoom Call Ends
     ↓
[Stage 1] Webhook fires → recording + transcript retrieved
     ↓
[Stage 2] AI processes transcript → structured summary
     ↓
[Stage 3] Action items extracted with assignees + due dates
     ↓
[Stage 4] CRM updated with meeting notes + client sentiment
     ↓

Let's walk through each one.

Stage 1: Capturing the Recording and Transcript

Everything starts with Zoom's webhook system. When a cloud-recorded meeting ends, Zoom fires a recording.completed event to our registered webhook endpoint. This payload includes the meeting ID, participant list, duration, and download URLs for both the recording and the auto-generated transcript.

Our system uses Zoom's Cloud Recording API with server-to-server OAuth to pull the transcript file. Zoom provides transcripts in VTT format with speaker identification and timestamps — this is the raw material the rest of the pipeline works with.

Key implementation detail: we don't rely on Zoom's built-in AI summary features. Instead, we pull the raw transcript and process it ourselves. This gives us full control over extraction logic, output format, and downstream integrations.

// Webhook handler (simplified)
app.post('/webhook/zoom', async (req, res) => {
  const { event, payload } = req.body;
  
  if (event === 'recording.completed') {
    const { meeting_id, recording_files } = payload.object;
    const transcript = recording_files.find(f => f.file_type === 'TRANSCRIPT');
    
    // Queue for async processing
    await queue.add('process-meeting', {
      meetingId: meeting_id,
      transcriptUrl: transcript.download_url,
      participants: payload.object.participant_audio_files
    });
  }
  
  res.status(200).send();
});

Stage 2: AI Transcript Processing

Raw transcripts are messy. Zoom's auto-transcription includes filler words, speaker misidentification, and crosstalk. Before we summarize, we run a cleaning pass:

  1. Speaker normalization — match Zoom's speaker labels to actual participant names using the meeting's participant list
  2. Segment merging — combine consecutive utterances from the same speaker into coherent blocks
  3. Noise removal — strip filler words, false starts, and off-topic small talk (while preserving context)

The cleaned transcript then goes to an LLM with a carefully tuned prompt. We don't just ask for "a summary." The prompt requests a structured output with specific fields:

{
  "executive_summary": "One paragraph, 3-4 sentences max",
  "key_decisions": ["Decision 1 with context", "Decision 2..."],
  "action_items": [
    {
      "description": "What needs to happen",
      "assignee": "Person who committed",
      "due_date": "Extracted or inferred deadline",
      "confidence": 0.92,
      "source_quote": "Exact words from transcript"
    }
  ],
  "open_questions": ["Unresolved items that need follow-up"],
  "sentiment": {
    "overall": "positive|neutral|negative",
    "client_satisfaction": 0.85,
    "risk_flags": ["Any concerns or blockers mentioned"]
  }
}

The confidence score is critical. Early on, we found the AI would flag casual remarks as action items ("I should probably look into that sometime"). By requiring a confidence threshold and the source quote, we can distinguish between real commitments and conversational fluff. Items below 0.7 confidence get flagged for human review instead of auto-assigned.

Stage 3: Action Item Extraction and Assignment

This is where most generic meeting tools stop — they give you a bullet list and call it a day. We go further:

  • Assignee matching — the AI extracts who committed to the action, then our system matches that name to a real user in our project management tool
  • Due date inference — if someone says "I'll have that by Friday," the system calculates the actual date. If no deadline is mentioned, it suggests one based on urgency signals in the conversation
  • Deduplication — if the same action item was discussed multiple times in the meeting, we merge them and keep the most specific version
  • Context linking — each action item carries a reference back to the exact moment in the transcript where it was discussed, so anyone can verify the original context

The extracted action items are written to a structured database that serves as the system of record. From here, they can flow to any downstream tool — Linear, Jira, Asana, a custom dashboard — via API.

Stage 4: CRM Update

For client-facing meetings, the pipeline includes a CRM integration step. The structured meeting data — summary, decisions, action items, and sentiment analysis — gets pushed to the relevant contact or deal record.

This solves a problem every services company knows: CRM notes are either empty or months out of date. When meeting intelligence flows automatically, your CRM becomes a living record of every client interaction.

The sentiment analysis is particularly valuable for account management. If a client expresses frustration in a meeting, that risk signal shows up on their account record immediately — not two weeks later when someone remembers to mention it in a standup.

Stage 5: Slack Notification

The final stage delivers the output where people actually work: Slack. Within 60 seconds of the meeting ending, the relevant channel receives a formatted message with:

  • Executive summary (2-3 sentences)
  • Key decisions made
  • Action items with owners and due dates
  • A link to the full meeting record in the dashboard

The notification is routed to the right channel automatically based on the meeting's calendar event — client meetings go to the client channel, internal standups go to the team channel, sales calls go to the pipeline channel.

Team members can react to action items directly in Slack to acknowledge them, or flag items that need correction. This closes the feedback loop without anyone leaving their primary workspace.

What Makes This Different from Otter, Fireflies, or Zoom AI Companion

There are dozens of meeting transcription products. Most of them do stages 1 and 2 well enough. Here's what they don't do:

  • Custom extraction logic — generic tools use one-size-fits-all prompts. Our pipeline is tuned to how our team talks about commitments, deadlines, and decisions
  • Deep integration — summary-to-CRM-to-Slack-to-project-tracker as a single automated flow, not manual copy-paste between tools
  • Confidence scoring — knowing which action items are rock-solid vs. tentative changes how you follow up
  • Institutional memory — a searchable archive of every decision made across every meeting, not siloed per-meeting summaries
  • Custom routing — output goes to the right people and channels automatically, based on meeting context

The difference isn't the AI model — it's the integration layer. Any LLM can summarize a transcript. The hard part is connecting that summary to the systems where work actually happens.

Building This Yourself

If you want to build a similar pipeline, here's the minimum viable stack:

  1. Zoom Server-to-Server OAuth app — for webhook events and API access to recordings/transcripts (Zoom S2S OAuth docs)
  2. Webhook endpoint — to receive recording.completed events
  3. Message queue — don't process synchronously; webhooks have timeout constraints
  4. LLM with structured output — Claude or GPT-4 with JSON mode for reliable extraction
  5. Integration layer — API connections to your CRM, Slack, and project management tools

The engineering isn't exotic. The value is in the prompt engineering for reliable extraction, the routing logic for getting output to the right place, and the feedback loops that let the system improve over time.
